# 02 — File & Folder Structure

> **Goal:** Understand what every single file in this project does, why it exists, and what happens if you remove it.

## Top-Level Layout

```
backend/
├── src/               ← All your application source code lives here
├── dist/              ← Compiled JavaScript output (auto-generated, never edit)
├── test/              ← End-to-end test files (optional)
├── docs/              ← This documentation
├── nest-cli.json      ← NestJS CLI config (tells Nest where src/ is)
├── tsconfig.json      ← TypeScript compiler options
├── tsconfig.build.json← TypeScript config used during `nest build`
├── package.json       ← npm dependencies and scripts
```

## Every File — Explained

### (1) `src/main.ts` — The Entry Point

**This is the very first file Node.js runs.** Everything starts here.

```typescript
async function bootstrap() {
  const app = await NestFactory.create(AppModule);  // create the app from root module
  app.setGlobalPrefix('api');                        // all routes: /api/...
  app.enableCors({ origin: configService.get('FRONTEND_URL') });
  app.useGlobalPipes(new ValidationPipe({ ... }));   // validate all request bodies
  SwaggerModule.setup('api/docs', app, document);    // serve API docs at /api/docs
  await app.listen(configService.get('PORT'));
}
bootstrap();
```

**What it does:**
- Creates the NestJS app using `AppModule` as the root
- Sets up global middleware: CORS, validation, Swagger
- Starts listening for HTTP requests

**What happens if you remove it?** The app never starts.

---

### (2) `src/app.module.ts` — The Root Module

**The root of the entire application.** This is what `main.ts` passes to `NestFactory.create()`.

```typescript
@Module({
  imports: [
    TypeOrmModule.forRoot({ ... }),  // database connection (set up ONCE here)
    AuthModule,                       // plug in auth feature
    TodoModule,                       // plug in todo feature
  ],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

**Key rule:** `TypeOrmModule.forRoot()` is called **once** in the root module to establish the database connection. Feature modules then use `TypeOrmModule.forFeature([EntityClass])` to access specific tables.

**What happens if you remove a module from `imports`?** That module's routes and services stop working entirely.

---

### (3) `src/app.controller.ts` — Root Controller

A minimal controller that handles the root route `/`. Used as a health check.

```typescript
@Controller()
export class AppController {
  @Get()
  getHello(): string {
    return this.appService.getHello();  // returns "Hello World!"
  }
}
```

**GET /** → `"Hello World!"`  
This is the default generated by the NestJS CLI. You can delete it once you have real routes.

---

### (4) `src/app.service.ts` — Root Service

The simplest possible service — just returns a string.

```typescript
@Injectable()
export class AppService {
  getHello(): string { return 'Hello World!'; }
}
```

## Why Does Each Module Have 3 Files?

When you look at `auth/` or `todo/`, you always see:

```
todo/
├── todo.module.ts      ← the module
├── todo.controller.ts  ← the controller
└── todo.service.ts     ← the service
```

This is the **standard NestJS pattern**. Here's why each file exists:

| File | Purpose | Analogy |
|------|---------|---------|
| `*.module.ts` | Wires controller + service together; declares what the module exposes | A department's org chart |
| `*.controller.ts` | Handles HTTP routes; reads request data; calls the service | The receptionist |
| `*.service.ts` | Business logic; database calls; throws errors | The actual employee doing work |

### The Rule: Controller never touches the database

Controllers only:
1. Extract data from requests (`@Body()`, `@Param()`, `@GetUser()`)
2. Call service methods
3. Return the result

Services only:
1. Contain business logic
2. Query the database via repositories
3. Throw appropriate HTTP exceptions

### (5) `auth/auth.module.ts`

Configures the authentication feature. Sets up JWT and Passport:

```typescript
@Module({
  imports: [
    PassportModule.register({ defaultStrategy: 'jwt' }),
    JwtModule.registerAsync({ inject: [ConfigService], useFactory: (...) }),
    TypeOrmModule.forFeature([User]),   // give AuthService access to the users table
  ],
  controllers: [AuthController],
  providers: [AuthService, JwtStrategy],
  exports: [JwtStrategy, PassportModule],  // share these with other modules
})
```

**Why `exports`?** The `TodoModule` needs to use `JwtAuthGuard`, which depends on `JwtStrategy` and `PassportModule`. By exporting them here, `TodoModule` can import `AuthModule` and use the guard without re-configuring JWT.

---

### (6) `auth/auth.controller.ts`

Exposes three public routes:

```
POST /api/auth/register  → calls authService.register()
POST /api/auth/login     → calls authService.login()
GET  /api/auth/profile   → calls authService.profile() [requires JWT]
```

Notice that `/register` and `/login` have **no guard** — they are intentionally public because you need to register/login to get a token in the first place.

---

### (7) `auth/auth.service.ts`

The brain of authentication:

- **`register()`** — checks email uniqueness → hashes password → saves user → returns JWT
- **`login()`** — finds user → compares bcrypt hash → returns JWT
- **`generateToken()`** — private helper that signs the JWT payload

```typescript
private generateToken(user: User): string {
  const payload = { sub: user.id, email: user.email };
  return this.jwtService.sign(payload);  // creates: "eyJhbGciOi..."
}
```

---

### (8) `auth/jwt.strategy.ts`

This file tells Passport **how to validate a JWT token** on every protected request.

Flow:
1. Request arrives with `Authorization: Bearer <token>`
2. Passport extracts the token from the header
3. Verifies the signature using the secret key
4. Decodes the payload: `{ sub: 1, email: "john@example.com" }`
5. Calls `validate(payload)` — we look up the user in DB
6. The returned user is attached to `request.user`

---

### (9) & (10) `auth/dto/register.dto.ts` and `login.dto.ts`

DTOs (Data Transfer Objects) define the **expected shape of incoming request data** and attach **validation rules** via decorators from `class-validator`.

```typescript
export class RegisterDto {
  @IsString()
  @IsNotEmpty()
  @MaxLength(100)
  name: string;

  @IsEmail()
  email: string;

  @MinLength(6)
  password: string;
}
```

The global `ValidationPipe` (set up in `main.ts`) automatically:
- Checks that the request body matches the DTO
- Returns `400 Bad Request` with error messages if it doesn't
- Strips unknown properties (`whitelist: true`)

---

### (11) `auth/guards/jwt-auth.guard.ts`

A guard is a **gatekeeper** for routes. This one extends NestJS's built-in `AuthGuard('jwt')`:

```typescript
@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {}
```

When you apply `@UseGuards(JwtAuthGuard)` to a route:
- NestJS runs the guard BEFORE the route handler
- If the JWT is missing/invalid → returns `401 Unauthorized` immediately
- If valid → attaches the user to `request.user` and continues

---

### (12) `auth/decorators/get-user.decorator.ts`

A custom **parameter decorator** that extracts `request.user` cleanly:

```typescript
// Without it (ugly):
@Get()
findAll(@Req() req: Request) {
  const user = req.user as User;  // not type-safe
}

// With it (clean and type-safe):
@Get()
findAll(@GetUser() user: User) {
  console.log(user.id);  // fully typed
}
```

### (16) `todo/dto/create-todo.dto.ts`

```typescript
export class CreateTodoDto {
  @IsString() @IsNotEmpty() @MaxLength(255)
  title: string;

  @IsString() @IsOptional()
  description?: string;
}
```

Only `title` is required. `description` is optional.

---

### (17) `todo/dto/update-todo.dto.ts`

```typescript
export class UpdateTodoDto extends PartialType(CreateTodoDto) {
  @IsBoolean() @IsOptional()
  isComplete?: boolean;
}
```

`PartialType(CreateTodoDto)` makes all fields from `CreateTodoDto` optional. This means a PATCH request can send just `{ "isComplete": true }` without needing to resend the title.

### (18) `todo/entities/todo.entity.ts`

Defines the `todos` database table. TypeORM reads this class and creates the table automatically (when `synchronize: true`).

```typescript
@Entity('todos')
export class Todo {
  @PrimaryGeneratedColumn() id: number;
  @Column() title: string;
  @Column({ nullable: true }) description: string | null;
  @Column({ name: 'is_complete', default: false }) isComplete: boolean;

  @ManyToOne(() => User, user => user.todos, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'user_id' })
  user: User;

  @Column({ name: 'user_id' }) userId: number;

  @CreateDateColumn({ name: 'created_at' }) createdAt: Date;
  @UpdateDateColumn({ name: 'updated_at' }) updatedAt: Date;
}
```

**Resulting SQL table:**
```sql
CREATE TABLE todos (
  id          SERIAL PRIMARY KEY,
  title       VARCHAR(255) NOT NULL,
  description TEXT,
  is_complete BOOLEAN DEFAULT false,
  user_id     INTEGER REFERENCES users(id) ON DELETE CASCADE,
  created_at  TIMESTAMP DEFAULT now(),
  updated_at  TIMESTAMP DEFAULT now()
);
```

### (19) `user/entities/user.entity.ts`

```typescript
@Entity('users')
export class User {
  @PrimaryGeneratedColumn() id: number;
  @Column({ length: 100 }) name: string;
  @Column({ unique: true }) email: string;

  @Column({ select: false })  // ← NEVER returned in queries by default
  password: string;

  @OneToMany(() => Todo, todo => todo.user)
  todos: Todo[];

  @CreateDateColumn({ name: 'created_at' }) createdAt: Date;
  @UpdateDateColumn({ name: 'updated_at' }) updatedAt: Date;
}
```

Key point: `select: false` on `password` means if you do `userRepository.find()`, the password hash is **never included** in the result — protecting you from accidentally leaking it in an API response.

## File Naming Conventions

NestJS follows strict naming conventions:

| Pattern | Example | Purpose |
|---------|---------|---------|
| `*.module.ts` | `todo.module.ts` | Module definition |
| `*.controller.ts` | `todo.controller.ts` | HTTP route handler |
| `*.service.ts` | `todo.service.ts` | Business logic |
| `*.entity.ts` | `todo.entity.ts` | Database table mapping |
| `*.dto.ts` | `create-todo.dto.ts` | Input validation shape |
| `*.guard.ts` | `jwt-auth.guard.ts` | Route protector |
| `*.strategy.ts` | `jwt.strategy.ts` | Passport auth strategy |
| `*.decorator.ts` | `get-user.decorator.ts` | Custom decorator |
